## 2. 本轮已完成的稳定底座

### 2.1 后端

- 已建立 `issue_fact` 共享事实读取与消费链路，避免系统测试、客户问题等模块重复维护事实层 SQL 和映射逻辑。
- 已将评审数据相关的大服务拆分为更清晰的 query / command / summary / option 职责。
- 已将系统测试与客户问题的核心统计板收口到共享的 issue-fact board runtime/support 能力上。
- 已把部分重复的指标、筛选、事实查询逻辑下沉到可复用底层，而不是继续散落在具体页面服务中。

### 2.2 前端

- 已完成 API 领域拆分，`api.ts` 不再承担全部能力实现，真实请求能力已下沉到 `api-client/*`。
- 已完成共享类型拆分，类型与 helper 已从旧聚合 API 文件中抽离。
- 已完成条件筛选状态收口，形成 `useConditionFilterGroupState`。
- 已完成条件筛选型记录页控制器收口，形成 `useRecordPageController`。
- 已完成规则说明侧边抽屉复用，形成 `useRuleExplanationPanel`。
- 已完成 `feature-manifest.ts` 收口，统一管理：
  - 模块与页面清单
  - route meta
  - query contract
  - statistic board page -> boardKey 映射
  - 特殊路由契约

## 3. 当前有意保留的特例

这些项目前不是“遗漏”，而是当前明确接受的特例，不影响解耦主线完成判断。

### 3.1 `ReviewDataManagementView`

- 仍保留较重的页面特例控制流。
- 原因是该页包含展开编辑、汇总加载、明细联动等复合交互，不适合为了统一而硬塞进通用记录页 controller。
- 结论：暂不强并，保持特例实现。

### 3.2 `SystemTestIssueSearchView`

- 仍使用旧式筛选模型，没有切到新一代条件筛选链路。
- 原因是该页查询模型与当前条件筛选抽象的重合度还不够高，强切会带来不必要耦合。
- 结论：先保留，后续若系统测试查询体系继续扩展，再评估迁移。

### 3.3 特殊路由

- `external/code-review-form`
- `code-review/illegal-records/rule-config`
- `not-found`

这些路由已经纳入 manifest 契约管理，但仍属于“非标准 shell 页面”，所以不与普通模块页面共用同一套路由生成模型。

## 4. 当前剩余问题的性质

本轮之后剩下的问题，已经不属于“解耦主线未完成”，而属于下面几类：

- 功能继续补齐类：如后续还要补的新模块、新页面、新看板。
- 数据口径补齐类：如真实数据未接入前的事实层命中率、业务口径完整度问题。
- 页面体验优化类：如构建包体过大、局部首屏加载体验、个别页加载策略优化。
- 特例继续收口类：仅在后续确认收益明确时，再逐步消化现存特例页面。

## 5. 后续开发约束

后续继续开发时，默认遵循以下规则：

1. 新增统计板页面时，优先接入现有 statistic-board runtime / manifest / base table，不再单独硬编码 board 映射。
2. 新增条件筛选型记录页时，优先复用 `useConditionFilterGroupState` 和 `useRecordPageController`。
3. 新增页面 route meta、query 白名单、persisted query、特殊路由例外时，统一先落到 `feature-manifest.ts`。
4. 新增 issue 事实型能力时，优先走事实层和共享查询层，不再回到页面服务里手写一套 `FACT_SQL + mapRecord`。
5. 若出现强交互页面无法复用通用 controller，应明确标记为“特例页面”，而不是静默绕开底座。

## 6. 建议的后续阶段命名

从 2026-04-23 起，建议不再把后续工作称为“解耦主线”，而改为以下三个方向：

1. 功能补齐阶段
2. 稳定化与体验优化阶段
3. 数据口径与真实链路完善阶段

这样可以避免把后续所有问题都混成“还没解耦完”。

## 7. 当前建议的下一步优先级

如果继续推进，建议按这个顺序：

1. 先补业务功能缺口，不动已经稳定的解耦底座。
2. 再做高收益体验优化，例如首屏加载、局部骨架、包体优化。
3. 最后在真实数据链路逐步接入后，再修正事实层口径和个别特例页面。
