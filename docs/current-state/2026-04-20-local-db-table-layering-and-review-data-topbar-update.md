# 本地库表分层与评审数据管理页顶部收口
更新时间：2026-04-20
适用项目：`D:\projects\data_collection_platform`

## 1. 目标

本轮主要收口两件事：

1. 把当前本地正式库里的表按职责分层，统一命名口径，避免后续继续把“业务表 / 事实表 / 同步支撑表”混在一起讨论。
2. 优化评审数据管理页顶部区域，去掉无用说明、页面标题和大块留白，让筛选和主表更快进入视野。

## 2. 本地库表分层

当前 `backend/src/main/resources/schema.sql` 中的正式表，建议按以下 `5` 类理解。

### 2.1 业务表

这类表承载平台自己的业务动作结果，不是 GitLab 原始镜像，也不是面向统计的统一事实层。

- `collect_form_records`
- `review_records`
- `review_record_experts`
- `review_problem_items`

说明：

- `collect_form_records` 用于表单采集结果落表。
- `review_records / review_record_experts / review_problem_items` 用于评审数据管理主记录、评审专家和问题清单。
- 这类表可以由表单提交、页面维护、后续 Excel 导入、镜像预填后补录等方式产生。

### 2.2 事实表

这类表承载归一化后的业务事实，主要服务查询、统计、规则判定和看板。

- `merge_request_fact`
- `issue_fact`

说明：

- 它们不直接代表页面录入对象。
- 它们主要来自镜像数据归一化和少量补充映射。
- 适合作为记录页、统计页、规则判定服务的统一读取层。

### 2.3 外部导入补充表

这类表用于给事实表补充外部来源指标，不直接等于主业务表。

- `code_review_external_metrics`

说明：

- 这类表适合接工具结果、外部系统结果、人工整理后的补充指标。
- 当前它主要补充代码走查场景中的注释率、缺陷数等外部指标。

### 2.4 配置 / 运维表

这类表承载同步配置、系统映射关系、日历配置等控制面数据。

- `gitlab_sync_configs`
- `testing_phase_calendar`
- `sys_table_registry`

说明：

- 这类表更多面向配置页和运维页。
- 它们控制“怎么同步、怎么注册、怎么解释阶段”，而不是承载业务明细。

### 2.5 运行支撑表

这类表承载任务、日志、Webhook、原始镜像快照等运行期支撑信息。

- `gitlab_sync_logs`
- `gitlab_sync_tasks`
- `gitlab_webhook_events`
- `gitlab_mirror_records`

说明：

- 这类表更适合后台诊断、补偿、审计、追踪。
- 不建议直接拿来当业务页或记录页的主数据源。

## 3. 对后续建模的统一判断

后面讨论新表时，建议先按这套分层判断：

- 用户录入、维护、导入后长期保存的正式业务对象：优先放业务表
- 统一供看板、规则、明细查询读取的归一化对象：优先放事实表
- 外部工具、机器人、补充指标结果：优先放外部导入补充表
- 同步配置、日历、映射：优先放配置 / 运维表
- 任务、日志、原始快照：优先放运行支撑表

这也意味着：

- 评审数据管理当前继续使用 `review_records / review_record_experts / review_problem_items` 是合理的
- 不建议把评审数据误叫成“事实表”
- 也不建议把导入结果直接混进运行支撑表或镜像快照表

## 4. 评审数据管理页顶部优化

文件：

- `frontend/src/views/ReviewDataManagementView.vue`

本轮对顶部区域做了减法，不调整功能，只收表现层：

- 去掉顶部大段解释文案
- 去掉页面标题 `评审数据管理`
- 去掉副标题 `主记录与问题清单统一维护`
- 让页面和其他记录页一样，从摘要区和表格区直接开始
- 压缩摘要卡与主表之间的垂直间距
- 缩小摘要卡高度与内边距，让筛选区和主表更早进入首屏

## 5. 当前效果

这次调整后的页面结构更接近：

- 紧凑摘要卡
- 直接进入筛选区和记录表

也就是说：

- 不再让顶部说明和标题抢占首屏
- 不再让用户先读一段文案再开始操作
- 更符合当前项目“Element Plus 基底 + 轻阿里系风格 + 低培训成本”的目标

## 6. 结论

当前本地正式库已经可以稳定按“业务表 / 事实表 / 外部导入补充表 / 配置运维表 / 运行支撑表”来讨论。

评审数据管理页顶部也已经按同样思路收口：

- 不改主流程
- 不加新功能
- 只压缩无效说明和无效留白
- 让真正高频的筛选与表格尽快进入用户视野
