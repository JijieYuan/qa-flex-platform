## 3. 系统测试模块复用结论

- 系统测试缺陷汇总
- 议题阶段统计

- `StatisticBoardView`

统一复用：

- `BaseRecordTable`
- 路由表格状态管理
- 记录列表接口 + 筛选项接口模式

## 4. 老平台客户问题模块评估结论

老平台“客户问题统计”菜单位于：

- `webapp/src/views/PageHome/Home.vue`
- `webapp/src/Router.js`

其菜单项为：

- 缺陷汇总
- 缺陷非法数据
- 缺陷原因分析
- `CC_PRODUCT` 议题
- 延期问题
- 缺陷响应效率
- 按功能展示缺陷数量

### 4.1 与系统测试明显同构的页面

这些页面不是业务上完全相同，但页面结构和交互模式高度一致，属于“同构不同口径”：

- `缺陷汇总`
  - 系统测试：`ModuleTable.vue`
  - 客户问题：`ModuleTableCCProduct.vue`
  - 差异主要在筛选与数据范围
- `缺陷非法数据`
  - 系统测试：`IllegalIssueSearch.vue`
  - 客户问题：`IllegalIssueSearchCCProduct.vue`
  - 差异主要在系统测试阶段与 milestone 口径
- `缺陷原因分析`
  - 系统测试：`ModuleAndCause.vue`
  - 客户问题：`ModuleAndCauseCCProduct.vue`
  - 表结构和下钻模式高度一致

### 4.2 代码层面存在同构，但老平台未作为正式菜单开放的页面

- `议题查询`
  - 系统测试：`IssueSearch.vue`
  - 客户问题：`IssueSearchCCProduct.vue`
  - 首页菜单中客户问题版查询入口被注释，说明老平台有实现尝试，但未作为正式入口保留

### 4.3 客户问题领域内更独立的专题页

这些页面不适合直接等同于系统测试模块中的现有页面：

- `CC_PRODUCT` 议题
- 延期问题
- 缺陷响应效率
- 按功能展示缺陷数量

它们更适合视为客户问题域下的专题分析页，而不是系统测试页面的简单拷贝。

## 5. 当前推荐架构结论

推荐继续保留两个一级模块：

- 系统测试
- 客户问题

但不要为客户问题再复制一套页面基础设施，而是采用：

- 相同页面抽象
- 不同数据域口径
- 不同默认筛选项和规则说明

也就是说：

- 模块层面分开
- 抽象层面复用
- 数据范围层面配置化

## 6. 客户问题模块后续落地方向

结合当前代码库，客户问题模块最适合沿用系统测试已经形成的两条抽象链路：

- 统计矩阵型页面复用 `BaseStatisticTable`
- 记录查询型页面复用 `BaseRecordTable`

当前优先建议先落的客户问题页面是：

- 缺陷汇总
- 缺陷原因分析
- 缺陷非法数据

原因：

- 与系统测试现有页面的结构最接近
- 最容易用现有抽象做“换数据域不换骨架”
- 对后续客户问题其他专题页能形成模板带动
