# 2026-05-21 内网基础测试问题解决方案

## 目标

基于内网基础测试反馈，对统计下钻、同步日志、同步状态、自动刷新、全量补偿、查询组件和数据镜像监控进行统一方案设计。

## 修复铁律

任何后续修复都必须遵守：修复某一个功能时，不得破坏、回退或误改其他已经验证通过的功能。这是本平台迁移阶段的铁律。

- 修复前必须确认影响范围，优先做局部修改。
- 修复后必须至少跑对应模块的针对性测试；涉及共享组件、共享服务或状态文案时，必须补充相关回归测试。
- 发现修复会改变其他模块行为时，必须先明确这是必要的设计变更，而不是顺手改动。
- 文案、状态、日志和按钮行为也属于功能契约，不能因为内部状态调整而给用户造成新的误导。

核心目标不是把内部状态简单翻译成中文，而是让用户看到的每个状态都准确表达平台实际行为：

- 不把“等待执行”说成“失败”。
- 不把“已合并到当前同步”说成“排队中”。
- 不把“部分表被策略性跳过”说成“部分成功”。
- 不把“当前同步尚未完成”说成“刷新失败”。
- 不让用户误以为同一数据源存在多个并发同步进程。

## 当前环境前提

- 内网访问入口：`172.22.10.115:18181`。
- 后端端口 `18080` 默认仅用于容器/本机链路，不作为用户入口。
- 内网数据源采用直连方式，不通过 Docker 网络访问 GitLab PostgreSQL。
- 内网真实源预计存在 CC、DGM 两类数据源，外网测试环境无法完全复现真实多源。
- GitLab System Hook 不包含 issue events，issue 创建/修改主要依赖增量同步、补偿扫描或后续新增的全量补偿同步兜底。
- 平台同一数据源同一时间只允许一个真实同步执行进程；其他请求应合并、复用、跳过或等待当前同步完成。

## 统一显示原则

### 内部状态与用户状态分离

后端可以继续保留内部状态机：

- `SUBMITTED`
- `QUEUED`
- `RUNNING`
- `RETRYING`
- `SUCCESS`
- `PARTIAL_SUCCESS`
- `FAILED`
- `TIMEOUT`
- `CANCELLED`

但前端普通用户视图不得直接暴露这些状态词，也不得机械翻译为“排队中”“部分成功”。

建议新增用户态展示字段，或在前端集中转换：

| 内部状态/动作 | 用户侧显示 | 说明 |
| --- | --- | --- |
| `QUEUED`，且无正在执行 run | `等待执行` | 只在运维详情中展示 |
| `QUEUED`，且已有同源 run 正在执行 | `等待当前同步完成` | 避免用户误解为多进程排队 |
| `REUSED_RUNNING` | `已合并到当前同步` | 表示不会创建新同步进程 |
| `REUSED_QUEUED` | `已合并到待执行同步` | 仅用于高级详情 |
| `PARTIAL_SUCCESS`，失败表为可跳过表 | `已完成，部分表已跳过` | 不是失败 |
| `PARTIAL_SUCCESS`，存在必须处理的失败表 | `需要处理` | 明确需要运维介入 |
| 用户刷新被全量同步覆盖 | `已展示当前可用数据` | 不使用“失败” |
| 无新数据写入 | `已检查，无新增数据` | 同步日志可见，业务页不弹提示 |

### 禁用误导性文案

面向用户的普通视图中避免出现：

- `排队中`
- `部分成功`
- `同步失败`，用于可预期的合并、等待、跳过场景
- `刷新失败`，用于当前同步尚未完成但页面已展示可用数据的场景
- `补偿同步`，用于用户主动触发的单表刷新
- `增量同步`，用于用户主动触发的单表刷新

这些词可以保留在高级诊断详情或数据库状态字段中，但需要有解释。

## 方案一：统计下钻统一支持 GitLab 链接

### 问题

统计类页面下钻表格中的议题编号没有超链接，用户无法从统计明细直达 GitLab。

### 方案

建立统一的下钻链接契约：

- 后端所有统计下钻接口返回 `gitlabUrl` 或 `issueWebUrl`。
- 如果已有统一服务 `GitlabIssueLinkService`，所有统计详情接口都复用它。
- 前端下钻表格统一识别以下字段：
  - `issueId`
  - `issueIid`
  - `issueNo`
  - `issueUrl`
  - `gitlabUrl`
- 如果存在 URL，议题编号显示为链接；如果不存在，显示普通文本。

### 多源规则

链接生成必须同时依赖：

- `sourceInstance`
- `projectId`
- `issueIid`
- 对应数据源配置中的 `gitlabWebBaseUrl`

不能用单一全局 GitLab URL 盲拼，避免 CC/DGM 多源串源。

### 验收

- 系统测试、客户问题、代码走查、集成测试、质量看板的统计下钻都能点击议题编号。
- CC/DGM 不同来源生成不同链接。
- 缺少项目路径或 GitLab URL 时不报错，只展示文本。

## 方案二：同步日志用户态语义重构

### 问题

最近同步日志展示“排队中”“部分成功”“补偿同步”等词，容易误导用户。

### 方案

同步日志分三层：

1. **运行类型**：用户触发或系统触发的同步目的。
2. **处理结果**：本次请求实际怎么处理。
3. **高级状态**：内部 run/table task 机器状态，只在展开详情中显示。

### 运行类型命名

| 内部 runType | 用户侧类型 |
| --- | --- |
| `FULL_SYNC` | 全量同步 |
| `INCREMENTAL_SYNC` | 刷新最新数据 |
| `TABLE_REFRESH` | 单表刷新 |
| `SYSTEM_HOOK` | System Hook 唤醒 |
| `COMPENSATION_SCAN` | 自动补偿扫描 |
| `FACT_REFRESH` | 事实数据刷新 |
| `FULL_COMPENSATION_SCAN` | 全量补偿对账 |

### 处理结果命名

| 场景 | 用户侧结果 |
| --- | --- |
| run 完成且无异常 | 已完成 |
| run 正在执行 | 正在处理 |
| 请求被活动 run 合并 | 已合并到当前同步 |
| 请求等待当前 run 完成 | 等待当前同步完成 |
| 没有发现变化 | 已检查，无新增数据 |
| 策略性跳过 | 已跳过，无需处理 |
| 需要运维处理 | 需要处理 |
| 用户取消 | 已取消 |

### 最近同步日志刷新

同步页面应对最近日志进行轮询：

- 同步页轮询间隔建议 3 秒。
- 当存在活动 run 时轮询 1 秒到 2 秒。
- 页面离开同步模块后停止高频轮询。
- 轮询不得重置表格分页、筛选、滚动位置。

### 验收

- 最近同步日志不再直接出现“排队中”“部分成功”。
- 单表刷新日志不再显示为“增量同步”或“补偿同步”。
- System Hook 接收后能在日志或监控里快速看到“已接收/已合并/正在处理”。

## 方案三：同步场景行为矩阵

### 统一规则

同一数据源同一时间只执行一个真实同步 run。后续请求只能：

- 合并到当前 run。
- 复用待执行 run。
- 记录为待当前同步完成后处理。
- 判断无必要执行并跳过。

不得创建同源并发同步执行。

### 场景矩阵

| 当前状态 | 用户动作/系统事件 | 后端处理 | 用户提示 | 日志显示 |
| --- | --- | --- | --- | --- |
| 空闲 | 点击全量同步 | 创建全量 run | 已开始全量同步 | 全量同步 / 正在处理 |
| 全量同步中 | 点击全量同步 | 不创建新 run | 全量同步正在进行 | 已合并到当前同步 |
| 全量同步中 | 点击刷新最新数据 | 不创建增量 run | 当前全量同步正在进行，已展示当前可用数据 | 已合并到当前同步 |
| 全量同步中 | 点击单表刷新 | 记录单表刷新意图或提示等待 | 当前全量同步完成后可刷新单表 | 单表刷新 / 等待当前同步完成 |
| 全量同步中 | 自动补偿扫描触发 | 跳过或合并 | 不打扰用户 | 自动补偿扫描 / 已跳过 |
| 增量同步中 | System Hook 到达 | 合并目标表 | System Hook 已接收 | System Hook 唤醒 / 已合并到当前同步 |
| 补偿扫描中 | 单表刷新 | 按优先级合并或等待 | 当前已有同步处理相关表 | 单表刷新 / 已合并到当前同步 |
| 任意同步中 | 修改自动同步开关 | 保存配置，下次生效 | 当前同步仍按启动时配置执行，新策略下次生效 | 配置变更 / 下次生效 |
| 任意同步中 | 修改白名单 | 保存配置，下次同步生效 | 当前同步使用启动时表范围，新白名单下次生效 | 白名单变更 / 下次生效 |

### 后端建议

同步提交接口返回结构应能表达真实处理结果：

```json
{
  "runId": 123,
  "requestedType": "TABLE_REFRESH",
  "effectiveType": "FULL_SYNC",
  "decision": "MERGED_TO_ACTIVE_RUN",
  "userStatus": "已合并到当前同步",
  "message": "当前全量同步正在处理该数据源，本次请求已合并，不会创建新的同步进程。"
}
```

这样前端不需要猜测 `QUEUED` 的真实含义。

## 方案四：单表刷新独立语义

### 问题

单表刷新当前会混入增量同步或补偿扫描语义。

### 方案

单表刷新必须作为独立用户动作：

- `requestedType = TABLE_REFRESH`
- 用户侧类型固定显示为 `单表刷新`
- 即使后端合并到全量 run，也显示 `单表刷新 / 已合并到当前同步`

不要把它改写成“增量同步”或“补偿同步”展示给用户。

### 日志建议

| 实际情况 | 用户侧日志 |
| --- | --- |
| 单表刷新独立执行 | 单表刷新 / 已完成 |
| 被全量同步覆盖 | 单表刷新 / 已合并到当前同步 |
| 当前表无基线 | 单表刷新 / 需先完成全量同步 |
| 当前表无变化 | 单表刷新 / 已检查，无新增数据 |

## 方案五：定时全量补偿同步

### 定义

定时全量补偿同步不是全量重建，也不是简单的每张表全量覆盖。

最终目标定义为：

> 在低峰期对全部白名单表执行源库与镜像库的完整对账：源库缺失的数据要从镜像侧标记删除或物理清理，镜像缺失的数据要补齐，字段值不一致的数据要按源库纠正，多余的镜像数据也要删除，最终保证镜像库与源库在可对账范围内一致。

当前实现状态：已补齐 `FULL_COMPENSATION_SCAN` 的核心执行语义、手动入口和每日执行时间配置。该运行按白名单表做全表分批扫描，源库行强制覆盖镜像行以纠正字段漂移；源库不存在但镜像仍存在的主键会被标记删除；该运行与全量、增量、单表刷新、System Hook 和普通补偿扫描保持同源互斥。后台调度器每分钟检查各数据源配置，只在到达对应“每日执行时间”时触发。

### 执行策略

- 配置项：
  - 已有：是否启用全量补偿对账。
  - 已有：每日执行时间，例如 `02:00`。
  - 已有：数据同步页面提供“全量补偿对账”手动入口，便于内网验收时主动触发。
  - 后续增强：最大执行时长、每轮最大表数或最大批次数在前端可配置。
- 与其他同步互斥，不能和全量/增量/单表刷新并行。
- 当前未做成独立大设置区：删除策略、限速策略和最近一次对账结果后续再细化；本轮先把启用开关、每日执行时间和立即执行能力放到同步设置区，避免空有内部类型但用户无法触发。

### 表级策略

| 表特征 | 处理方式 |
| --- | --- |
| 有主键 + updated_at | 按主键全量对账；新增/变更行按源库写入，源库不存在的镜像行按策略删除或标记删除 |
| 有主键无 updated_at | 按主键窗口全量扫描；逐窗口比对源库与镜像库的存在性和字段摘要 |
| 无可靠主键 | 只做行数/摘要诊断，不自动删除或重写，避免误删 |
| 大表 | 限速、分片、可中断 |

### 用户侧名称

手动入口显示为 `全量补偿对账`，日志中按触发原因区分“手动全量补偿对账”和“定时全量补偿”。不要显示为 `全量同步`，避免用户误以为平台在重建所有数据。

## 方案六：同步中调整策略

### 结论

允许用户在同步进行中修改策略，但必须明确“当前 run 使用启动时配置，新策略下次生效”。

### 方案

- 后端在 run 创建时保存配置快照：
  - 白名单模式
  - 表范围
  - 线程数
  - 补偿间隔
  - 自动同步开关
- 正在执行的 run 不读取实时配置变更。
- 配置保存接口正常保存新策略。
- 前端如果检测到当前数据源有活动 run，保存后展示：

```text
设置已保存。当前同步仍按启动时配置执行，新设置将在下一次同步生效。
```

### 禁止误导

不要显示：

- `设置失败`
- `同步失败`
- `配置未生效`

除非保存接口真的失败。

## 方案七：业务页面自动刷新最新数据

### 需求解释

用户进入业务页面时，希望看到最新数据。但这不应该等同于每次进入页面都触发 GitLab 源库同步。

### 方案

分成两个动作：

1. **页面数据刷新**：重新查询平台事实表或镜像表，低成本，默认可开启。
2. **源库追数同步**：触发增量/补偿，成本较高，只能由同步策略控制。

### 推荐实现

- 增加用户设置：`进入业务页面时自动刷新页面数据`。
- 默认开启。
- 只刷新当前页面数据，不触发 GitLab 同步。
- 如果后台同步产生新写入，显示轻提示：

```text
检测到新数据，可刷新当前页面。
```

- 如果没有新写入，不提示用户。

### 禁止行为

- 不弹频繁 toast。
- 不触发全局 loading。
- 不清空表格。
- 不重置筛选、排序、分页。

## 方案八：刷新最新数据的提示语义

### 问题

用户点击“刷新最新数据”时，如果当前全量同步未结束，页面显示“失败”过于绝对。

### 方案

`刷新最新数据` 应改成“请求当前页面可用的最新结果”，不承诺一定触发同步。

### 状态文案

| 后端结果 | 用户提示 |
| --- | --- |
| 页面数据已刷新 | 已刷新当前页面数据 |
| 后台同步正在进行 | 当前同步仍在进行，已展示当前可用数据 |
| 请求被合并 | 当前已有同步在处理，本次刷新已合并 |
| 没有新数据 | 已检查，无新增数据 |
| 真实接口错误 | 刷新未完成，请稍后重试 |

只有网络错误、后端异常、权限错误等真实失败才显示“刷新未完成”。

## 方案九：系统测试议题查询条件重构

### 问题

系统测试议题查询中“模块名称”字段混入模块、里程碑、轮次等信息。

### 方案

采用统一条件查询组件，支持：

- 满足全部
- 满足任意

字段拆分为：

- 模块名称
- 测试阶段/轮次
- 里程碑
- 所属项目
- 严重程度
- 状态
- 标题关键词

如果后端事实表当前没有拆分字段，则先改前端文案为更准确的 `综合关键词`，避免继续误导用户。

## 方案十：数据镜像监控 UI 简化

### 速率显示

写入速率只保留整数：

- `1234 行/秒`
- 小于 1 时显示 `低于 1 行/秒`

不显示长小数。

### 当前运行摘要

默认展示：

- 同步类型
- 当前状态
- 已处理表数 / 总表数
- 写入记录数
- 耗时
- 当前处理表
- 最近消息

内部字段折叠到“高级详情”：

- runId
- workerId
- leaseUntil
- heartbeatAt
- runAfter
- raw status
- raw runType

### 最近同步日志

日志模块应升级为同步运维主入口：

- 自动刷新。
- 支持按类型筛选。
- 支持按结果筛选。
- 支持查看合并/跳过原因。
- 支持展开表级任务。
- 默认隐藏内部状态词。

## 方案十一：实现顺序

### 第一阶段：消除误导显示

1. 前端集中转换同步状态，不再直接展示 `QUEUED/PARTIAL_SUCCESS`。
2. 最近同步日志类型改为用户侧类型。
3. 单表刷新独立显示为 `单表刷新`。
4. 刷新最新数据不再把可预期等待显示为失败。

### 第二阶段：补齐链接和查询体验

1. 统计下钻统一支持 GitLab 链接。
2. 系统测试议题查询字段拆分或改为综合条件查询。
3. 数据镜像监控 UI 简化。

### 第三阶段：同步策略增强

1. 同步场景矩阵落到后端返回结构。
2. 同步中修改策略改为“保存成功，下次生效”。
3. 最近同步日志自动刷新。
4. 业务页面进入自动刷新页面数据。

### 第四阶段：定时全量补偿

1. 增加配置项。
2. 增加调度器。
3. 增加表级校验策略。
4. 增加源库压力保护。
5. 内网低峰窗口验证。

## 验收清单

- 普通用户视图不再出现“排队中”“部分成功”。
- 同一数据源任何时刻不出现多个实际执行中的同步进程。
- 用户点击任何同步相关按钮后，看到的是该请求真实处理结果。
- 统计下钻议题编号可以跳转 GitLab。
- System Hook 触发后同步日志能及时体现“已接收/已合并/正在处理”。
- 单表刷新日志始终显示为“单表刷新”。
