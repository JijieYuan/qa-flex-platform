# 数据库查看模块设计

**目标**

在系统设置左侧菜单中新增一个“数据库查看”页面，用于浏览本地平台数据库中的核心业务表数据。第一版只提供简单、稳定、低学习成本的浏览能力，不引入 SQL 编辑器、数据编辑、删除或复杂分析能力。

**范围**

- 仅查看本地数据库，不连接 GitLab 源库
- 页面入口位于“系统设置”模块左侧菜单
- 使用 Element Plus 原生表格能力
- 支持：
  - 表选择
  - 简单关键字筛选
  - 表格列排序
  - 分页
  - 手动刷新
- 不支持：
  - 任意 SQL
  - 新增/编辑/删除数据
  - 多条件复杂筛选器
  - 导出

## 方案选择

### 方案 1：系统设置下新增独立“数据库查看”页面

这是本次采用方案。

特点：
- 保持“同步设置”和“数据浏览”职责分离
- 页面结构最简单，符合当前系统设置布局
- 后续若要扩展“数据库查看”能力，不影响同步设置页

### 方案 2：把数据库查看塞进现有数据镜像设置页

不采用。

原因：
- 同步配置和数据浏览混在一起
- 页面会继续膨胀
- 不利于后续维护

### 方案 3：一开始就做通用数据表浏览器平台

本次不采用。

原因：
- 目标过重
- 超出当前“简单查看”的边界

## 页面设计

新增菜单项：
- 系统设置
  - 数据镜像设置
  - 数据库查看
  - 模块管理

页面结构：

1. 顶部工具栏
- 表选择 `el-select`
- 关键字搜索 `el-input`
- 刷新按钮 `el-button`

2. 表格区
- 使用 `el-table`
- 根据后端返回列动态渲染
- 开启原生列排序
- 内容过长时使用 `show-overflow-tooltip`

3. 分页区
- 使用 `el-pagination`
- 支持页码切换
- 支持每页条数切换

4. 空态与错误态
- 无数据时使用 `el-empty`
- 请求失败时使用页面内提示 + `ElMessage`

## 后端设计

新增一个轻量数据库浏览控制器，建议路径：
- `GET /api/database-browser/tables`
- `GET /api/database-browser/rows`

### 接口 1：表列表

返回当前允许浏览的本地表名列表。

建议第一版只开放白名单表，例如：
- `gitlab_sync_configs`
- `gitlab_sync_tasks`
- `gitlab_sync_logs`
- `gitlab_webhook_events`
- `gitlab_mirror_records`

原因：
- 避免前端任意探测数据库
- 方便后续权限和展示控制

### 接口 2：表数据

请求参数：
- `tableName`
- `page`
- `size`
- `keyword` 可选
- `sortField` 可选
- `sortOrder` 可选

返回内容：
- `columns`
- `rows`
- `total`
- `page`
- `size`

## 查询策略

第一版不做复杂字段级筛选推断。

统一策略：
- 关键字搜索按每张表预设的“可搜索列”执行
- 排序字段按白名单字段校验
- 未命中白名单字段时拒绝排序

这样可以保证简单稳定，不需要为每张表做复杂元数据推导。

## 安全边界

- 不允许前端直接传 SQL
- 表名必须在后端白名单中
- 排序字段必须在表字段白名单中
- 分页参数限制最大值，避免一次性拉过多数据

## 复用策略

虽然本次是系统设置下的独立页面，但实现上应拆成两层：

1. 后端
- 一个轻量通用服务：表元数据 + 分页查询

2. 前端
- 一个轻量通用组件：数据库表浏览器

这样后续别的模块如果也要简单看表数据，可以复用，而不是重写。

## 测试策略

### 后端
- 单元测试：
  - 非白名单表拒绝访问
  - 非白名单字段拒绝排序
  - 关键字搜索返回正确分页结果
- 集成测试：
  - 能返回表列表
  - 能返回动态列和分页数据

### 前端
- 组件测试：
  - 选择表后加载数据
  - 切换分页会重新查询
  - 排序会带上排序参数
- 构建验证：
  - `npm run build`

## 本次完成标准

- 系统设置左侧出现“数据库查看”
- 页面可切换查看本地数据库白名单表
- 可进行简单搜索、排序、分页、刷新
- 全部基于 Element Plus 原生组件
- 不引入复杂交互和额外业务逻辑
