# 2026-05-21 内网基础测试同步与体验问题清单

## 背景

内网基础测试已能启动平台并进入真实验证阶段。当前反馈集中在两类问题：

- 统计下钻、查询组件、同步监控等页面存在体验或功能缺口。
- 同步模块的实际状态、日志用语和用户感知之间存在偏差，容易让用户误判“失败”“排队”“部分成功”等状态。

本文只记录问题、调研方向和初步方案，不直接代表已完成修复。

## 当前发现的问题

### INTRANET-UX-001：统计类下钻表格缺少 GitLab 跳转链接

- **现象**：所有统计类表格的下钻明细中，议题编号没有赋予 GitLab 超链接功能。
- **影响**：用户从统计明细定位到具体 GitLab issue/MR 时需要手动搜索，影响排查效率。
- **期望**：所有统计类下钻表格中的议题编号都应复用统一的 GitLab 链接生成能力。
- **验收**：
  - 议题编号可点击。
  - 链接指向对应数据源配置的 GitLab Web 地址。
  - 多源场景下，CC/DGM 或其他 sourceInstance 不串源。

### INTRANET-SYNC-001：System Hook 唤醒到最近同步日志展示存在延迟感

- **现象**：System Hook 触发后，需要等待一段时间才会在最近同步日志中看到记录。
- **影响**：用户容易误以为 System Hook 没有生效。
- **初步判断**：
  - System Hook 本身可能已被后端接收，但日志列表刷新不够及时。
  - 同步 run 进入队列、执行、日志列表轮询之间缺少清晰的轻量状态反馈。
- **期望**：
  - System Hook 接收成功后，至少在同步日志或监控模块中快速看到“已接收/待处理/已合并”等明确状态。
  - 最近同步日志不依赖用户手动刷新整个页面。

### INTRANET-SYNC-002：同步状态“排队中”容易误导

- **现象**：同步模块经常出现“排队中”状态。
- **影响**：用户会疑惑：平台明明只允许同一时间一个同步进程，为何还会排队。
- **需要澄清的语义**：
  - “同一时间只允许一个同步进程”指同一数据源不会并行执行多个实际同步 worker。
  - “排队中”可能只是请求被记录、等待当前同步结束、或被合并复用，不等于真的有多个同步进程排队执行。
- **期望**：
  - 不再直接向普通用户展示“排队中”作为主状态。
  - 根据真实语义改成更准确的状态，例如：
    - `等待当前同步完成`
    - `已合并到当前同步`
    - `已接收，待系统处理`
    - `无需执行，当前数据已是最新`

### INTRANET-SYNC-003：单独表格刷新日志类型和结果用语误导

- **现象**：
  - 单独表格刷新在日志中有时显示为“增量同步”。
  - 在全量同步图中触发单独表格刷新后，完成全量同步时该记录可能显示为“补偿同步 / 部分成功”。
- **影响**：用户无法判断自己触发的是“单表刷新”“补偿扫描”还是“增量同步”，并可能误以为同步失败。
- **期望**：
  - 单独表格刷新必须有独立、稳定的日志类型，例如 `单表刷新`。
  - 如果因为全量同步正在运行而被合并、跳过或延后，应展示真实语义，而不是归类为补偿同步。
  - 不应使用“部分成功”描述用户不可行动、且系统已按策略处理的情况。

### INTRANET-SYNC-004：最近同步日志刷新能力不足

- **现象**：最近同步日志有时需要刷新整个页面才能看到最新记录。
- **影响**：同步模块作为运维观察入口，日志滞后会放大用户对同步状态的误解。
- **期望**：
  - 最近同步日志支持稳定轮询或事件式刷新。
  - 刷新不重置用户当前查看位置、筛选、展开项。
  - 日志状态应面向业务语义，而不是直接暴露内部状态机词汇。

### INTRANET-SYNC-005：页面状态栏对“刷新最新数据”的失败提示过于绝对

- **现象**：全量同步进行中，用户点击业务页面的“刷新最新数据”，页面状态栏可能显示同步失败。
- **问题**：功能并非真正失败，只是因为全量同步尚未完成，只能保证“当前已完成同步水位内”的最新数据。
- **期望提示**：
  - 不建议显示“失败”。
  - 可显示更准确的短提示，例如：
    - `当前全量同步仍在进行，已展示当前可用的最新数据`
    - `刷新请求已记录，将在当前同步完成后继续更新`
    - `当前数据已刷新到最近完成的同步批次`

### INTRANET-QUERY-001：系统测试议题查询模块筛选字段语义混乱

- **现象**：系统测试的议题查询模块中，“模块名称”字段里不只有模块名称，还混入里程碑、轮次等信息。
- **影响**：用户按模块查询时结果不可预期。
- **方案方向**：
  - 如果确实需要同时支持模块、里程碑、轮次等字段，应切换为支持“满足全部 / 满足任意”的条件查询组件。
  - 如果该字段只应表示模块名称，则需要清理数据源字段映射和前端筛选项文案。
- **验收**：
  - 用户能明确知道每个条件查询的是哪个字段。
  - 多条件组合时，能选择“全部满足”或“任意满足”。

### INTRANET-UI-001：数据镜像监控 UI 需要简化

- **现象**：
  - 线程写入速率显示过多小数位。
  - 当前运行详情展示了过多内部字段，即使研发也不一定能快速理解。
- **期望**：
  - 写入速率只保留整数部分。
  - 当前运行详情改成面向排查的摘要，例如：
    - 同步类型
    - 当前阶段
    - 已处理表数 / 总表数
    - 写入记录数
    - 耗时
    - 当前正在处理的表
  - 内部 runId、lease、worker 等字段可折叠到“高级详情”中。

## 新增功能需求

### FEATURE-SYNC-001：定时全量补偿同步

- **需求**：新增一种“全量补偿同步”，可配置每天固定时间执行，例如凌晨 2 点。
- **关键边界**：
  - 它不是重新执行一次真正的全量同步。
  - 它应以“全量对比 / 全表水位校验 / 差异补偿”为目标，尽量避免大规模重写。
  - 需要明确和现有全量同步、增量同步、补偿扫描的区别。
- **初步语义建议**：
  - `全量同步`：建立或重建镜像基线。
  - `增量同步`：按更新时间或水位追最新变化。
  - `补偿扫描`：短周期兜底，修复增量遗漏。
  - `全量补偿同步`：低频、定时、覆盖更广的差异校验，不重建全部数据。
- **待调研**：
  - 每张表是否都有可靠主键和更新时间字段。
  - 对没有更新时间字段的表，是否只能做主键窗口扫描或行数/校验摘要。
  - 对 GitLab 源库压力的影响。

### FEATURE-SYNC-002：同步进行中是否允许调整自动同步策略

- **需求**：用户希望同步进行时也能设置是否自动同步、补偿间隔等策略。
- **现状疑问**：当前平台可能会自动刷新回同步开始前保存的状态，导致用户在同步中修改策略被覆盖或看起来不生效。
- **待调研**：
  - 同步运行时读取的是 run 快照配置，还是实时读取配置表。
  - 前端状态刷新是否覆盖了用户未保存或刚保存的设置。
  - 是否允许“当前 run 使用旧配置，下一次 run 使用新配置”。
- **建议交互**：
  - 允许用户修改策略，但弹窗提示：`当前同步仍按启动时配置执行，新策略将在下一次同步生效。`
  - 如果某些参数会影响当前 run 安全性，则禁用并提示原因。

### FEATURE-PAGE-001：业务页面进入时自动刷新最新数据

- **需求**：用户每次进入除数据同步页面外的任意业务页面时，自动应用一次“刷新最新数据”。
- **可配置性**：该能力应允许用户自己开启或关闭。
- **约束**：
  - 不能造成页面抖动。
  - 不能在每个页面进入时都触发真实同步，优先理解为刷新当前页面数据或请求已有最新事实层结果。
  - 如果需要触发同步，必须遵守同源互斥、冷却和合并规则。
- **建议**：
  - 默认只做页面数据刷新，不触发 GitLab 源库同步。
  - 若检测到后台同步有新写入，再提示用户刷新当前页。

### FEATURE-SYNC-003：同步日志状态语义重构

- **需求**：同步日志不要出现容易误导的“部分成功”“排队中”等状态。
- **建议状态分层**：
  - 内部状态：继续保留 `QUEUED/RUNNING/PARTIAL_SUCCESS` 等机器状态，便于排障。
  - 用户状态：转换为中文业务语义。
- **用户侧状态建议**：
  - `已完成`
  - `正在同步`
  - `等待当前同步完成`
  - `已合并到当前同步`
  - `已跳过，无需同步`
  - `已完成，部分表已跳过`
  - `需要处理`
- **原则**：
  - 只有需要用户或运维介入时，才使用明显的异常状态。
  - 系统按预期合并、跳过、等待，不应显示为失败或部分失败。

### FEATURE-SYNC-004：同步模块场景总表

- **需求**：整理同步模块所有可能发生的情况，形成统一行为矩阵。
- **至少覆盖**：
  - 全量同步进行中点击全量同步。
  - 全量同步进行中点击增量同步。
  - 全量同步进行中点击补偿扫描。
  - 全量同步进行中点击单表刷新。
  - 全量同步进行中业务页面点击刷新最新数据。
  - 增量同步进行中触发 System Hook。
  - 补偿扫描进行中触发单表刷新。
  - 自动补偿扫描与用户手动同步冲突。
  - 源配置修改时已有同步正在运行。
  - 白名单修改时已有同步正在运行。
- **输出目标**：
  - 每个场景明确：是否创建 run、是否合并、是否跳过、日志显示什么、页面提示什么、是否需要用户操作。

## 优先级建议

| 优先级 | 编号 | 原因 |
| --- | --- | --- |
| P0 | INTRANET-SYNC-003 | 日志类型和结果误导会直接影响内网验收判断 |
| P0 | FEATURE-SYNC-003 | “部分成功/排队中”属于高频用户误解点 |
| P1 | INTRANET-UX-001 | 统计下钻无法跳 GitLab，影响所有统计页面排查效率 |
| P1 | INTRANET-SYNC-004 | 最近同步日志不及时会放大所有同步状态误解 |
| P1 | FEATURE-SYNC-004 | 需要先统一同步场景语义，再避免继续局部修补 |
| P1 | INTRANET-SYNC-005 | 业务页面状态提示过于绝对，影响用户信任 |
| P2 | FEATURE-SYNC-001 | 定时全量补偿同步是稳定性增强，需调研源库压力 |
| P2 | FEATURE-SYNC-002 | 同步中调整策略需要配置快照语义支持 |
| P2 | FEATURE-PAGE-001 | 页面自动刷新需谨慎，避免引入抖动和源库压力 |
| P2 | INTRANET-QUERY-001 | 查询语义混乱影响系统测试模块可用性 |
| P3 | INTRANET-UI-001 | UI 简化影响体验，但不阻断主链路 |

## 下一步建议

1. 先实现同步日志和状态语义重构，避免继续出现“排队中”“部分成功”等误导性展示。
2. 修复统计下钻议题编号链接能力，确保所有统计抽象类下钻表格统一支持 GitLab 跳转。
3. 梳理同步场景总表，并用它反推后端 run 合并、跳过、等待的用户侧状态。
4. 再设计定时全量补偿同步，先做源库压力评估和执行窗口约束。
5. 最后优化自动刷新与监控 UI，避免因为体验功能影响同步主链路稳定性。
