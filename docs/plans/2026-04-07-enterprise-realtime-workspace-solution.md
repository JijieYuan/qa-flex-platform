# 数据采集平台企业级近实时方案

## 1. 背景与现状

当前平台已经具备以下基础能力：

- GitLab 原始数据已通过同步链路进入本地镜像库
- 记录表抽象与统计表抽象已经拆分
- `代码走查非法记录` 与 `系统测试缺陷汇总` 已连通真实镜像查询
- 页面层已经具备“首屏先展示、后台再刷新”的近实时雏形

但从企业级中后台，尤其是阿里系控制台的标准来看，当前方案仍然偏“联调可用版”，主要问题是：

1. 页面查询与数据刷新边界刚刚建立，还没有完全平台化
2. 内网网络差，不能把“实时刷新”压在首屏请求里
3. 统计页仍偏向实时聚合，大数据量下会越来越重
4. 近实时能力需要作为页面工作区能力统一复用，而不是散落在各个页面里

本方案用于把现有实现收敛成一套适合当前平台阶段、同时兼顾未来扩展的正式方案。

## 2. 设计目标

### 2.1 用户体验目标

- 首屏必须快开，不能因为同步动作白屏或长时间等待
- 页面默认可用，刷新失败时旧数据仍然可看
- 用户能够明确知道当前数据是否最新
- 在弱网环境下，页面仍保持稳定与可预期

### 2.2 架构目标

- 表格抽象类保持纯粹，不承载同步逻辑
- 近实时能力以统一工作区壳的形式复用
- 记录页优先读取镜像/明细结果
- 统计页逐步过渡到结果层/快照层
- 外部数据统一走镜像同步链路，避免页面直接依赖外部系统

## 3. 推荐架构

### 3.1 四层结构

#### A. 原始镜像层 ODS

职责：

- 承接 GitLab 与外部平台进入 GitLab 后的原始数据
- 以 `ods_gitlab_*` 为核心保留原始事实
- 作为平台唯一可信原始来源

特点：

- 尽量少加工
- 保持可追溯
- 适合做后续所有业务层的基础输入

#### B. 业务明细层

职责：

- 为记录页提供稳定、可直接查询的业务记录模型
- 屏蔽多表联查、标签映射、派生字段等复杂逻辑

适用页面：

- 代码走查非法记录
- 代码走查明细
- 异常数据校验
- 统计下钻明细

建议形态：

- 业务宽表
- 视图
- 物化视图
- 中间结果表

#### C. 统计结果层

职责：

- 为统计表、排行榜、趋势图、看板卡片提供聚合结果
- 降低页面每次查询时的全量聚合成本

适用页面：

- 系统测试缺陷汇总
- 客户问题统计
- 开发者质量排行
- 代码走查合规统计

建议形态：

- 结果表
- 快照表
- 周期性重算结果

#### D. 页面工作区层

职责：

- 统一承载实时状态、刷新控制、失败降级、静默更新体验
- 不进入表格抽象类本体

典型能力：

- 最近同步时间
- 当前刷新状态
- 静默刷新
- 手动刷新
- 刷新失败后的降级提示

## 4. 页面读取与刷新策略

### 4.1 默认读取策略

页面默认只读取：

- 镜像表
- 业务明细结果
- 统计结果表或快照结果

禁止在首屏查询请求内部强依赖同步动作。

### 4.2 刷新策略

页面统一采用阿里系近实时方案：

1. 首屏直接展示当前已有结果
2. 页面加载后静默触发一次刷新
3. 刷新成功后无感更新页面
4. 刷新失败时保留旧数据，并提示“当前展示最近一次成功同步结果”
5. 用户可显式点击“刷新最新数据”

### 4.3 状态展示策略

页面右上工具区统一展示：

- 最近同步时间
- 当前状态：已同步 / 刷新中 / 刷新失败
- 刷新最新数据按钮

失败场景才显示明显告警；正常场景只保留轻量状态块，避免页面顶部过重。

## 5. 与表格抽象类的边界

### 5.1 不进入表格抽象类

以下能力不进入：

- `BaseRecordTable.vue`
- `BaseStatisticTable.vue`

原因：

- 这些能力不属于表格渲染本体
- 不是所有表格都需要实时刷新
- 强行塞入会污染抽象类职责

### 5.2 统一放入工作区壳

新增并统一复用：

- `RealtimeDataShell.vue`

职责：

- 状态展示
- 刷新控制
- 静默刷新
- 失败降级

这样记录页和统计页都能复用同一套交互策略，同时保持表格抽象本体稳定。

## 6. 当前阶段的适配方案

### 6.1 记录页

继续采用：

- 直接读取镜像/业务明细结果
- 页面静默刷新
- 用户手动刷新兜底

优先适配：

- 代码走查非法记录
- 后续其他记录类页面

原因：

- 记录页是分页查询，压力相对可控
- 用户更关心快开与能否看到变化

### 6.2 统计页

当前阶段保留：

- 首屏读取当前统计结果
- 页面静默刷新镜像并重新拉取 board 数据

中期必须演进：

- 统计结果表
- 统计快照表

原因：

- 统计页最容易在大数据量下变重
- 大量 issues、labels、label_links 联查与聚合不能长期放在页面实时请求里

### 6.3 外部数据

外部注释率、缺陷数等数据统一走：

外部平台 -> GitLab 侧 -> 镜像同步 -> 平台读取

不建议页面直接访问外部平台，原因是：

- 口径不统一
- 弱网环境不稳定
- 页面链路过长

## 7. 弱网专项策略

由于内网网络差，必须默认启用以下策略：

- 首屏查询绝不阻塞在同步动作上
- 刷新动作异步执行
- 刷新失败时旧数据继续可用
- 状态轮询轻量化，避免高频请求
- 刷新任务去重，避免用户连续点击造成任务堆积
- 首屏请求数尽量少，减少筛选项与列表的无效并发

## 8. 分阶段落地路线

### 第一阶段：统一近实时工作区能力

- 建立统一工作区壳
- 为记录页与统计页接入统一状态与刷新能力
- 保证首屏快开

### 第二阶段：整理业务明细层

- 将记录页依赖的派生字段、标签映射、业务规则逐步沉入业务明细层
- 降低页面临时拼装逻辑

### 第三阶段：引入统计结果层

- 为高成本统计页建立结果表/快照表
- 将大聚合从页面实时查询转移到后台计算

### 第四阶段：形成统一平台模式

- 原始镜像层
- 业务明细层
- 统计结果层
- 页面工作区壳

实现平台级统一复用。

## 9. 结论

适合当前平台的企业级方案不是“绝对实时”，而是：

- 记录页近实时
- 统计页结果化
- 页面先展示、再刷新
- 状态透明
- 失败可降级

这套方案同时兼顾了：

- 当前已经实现的真实链路
- 未来数据量增长
- 弱网内网环境
- 用户对“尽可能实时”的感知需求

它也是当前阶段最适合你们平台落地的阿里系/企业级方案。
