# 客户问题模块复用实施计划

## 1. 目标

客户问题模块后续实现采用“模块分开、抽象复用、数据域配置区分”的方式推进。

核心原则：

- 保留 `系统测试` 与 `客户问题` 两个一级模块
- 不为客户问题重复建设表格页面基础设施
- 尽量复用已经在系统测试模块落稳的统计板与记录表抽象
- 将差异尽量收敛到数据范围、默认筛选项、规则说明和文案配置

## 2. 复用分层方案

### 2.1 统计矩阵页模板

复用链路：

- `StatisticBoardPage`
- `StatisticBoardView`
- `BaseStatisticTable`

适用客户问题页面：

- 缺陷汇总
- 缺陷原因分析

### 2.2 记录查询页模板

复用链路：

- `BaseRecordTable`
- 路由表格状态
- 列表接口 + 筛选项接口

适用客户问题页面：

- 缺陷非法数据
- 后续如需恢复客户问题议题查询，也可继续复用

### 2.3 专题页模板

适用客户问题页面：

- `CC_PRODUCT` 议题
- 延期问题
- 缺陷响应效率
- 按功能展示缺陷数量

这类页面不建议硬塞进现有系统测试统计板定义中，但仍应复用：

- 通用筛选组件
- 基础表格组件
- 路由状态
- 导出与详情交互能力

- 因此客户问题查询能力可以视业务需要决定是否恢复，不纳入当前主计划第一批

## 4. 数据域策略

客户问题模块不应直接复用系统测试页面对应的 board key 或查询接口，而应复用其实现骨架并切换数据域。

建议后续统一引入“范围策略”概念，例如：

- `SystemTestScopeProfile`
- `CustomerIssueScopeProfile`

每个 profile 只负责回答：

- 当前模块的数据范围如何界定
- 默认筛选项是什么
- 是否按 `testingPhase`、`milestone`、`CC_PRODUCT` 或项目范围过滤
- 规则说明文案如何表达

这样可以做到：

- 前端页面模板不分叉
- 后端统计逻辑不重复铺开
- 模块差异集中在 profile 和 service 配置层

## 5. 实施顺序建议

### 第一批

- 客户问题缺陷汇总
- 客户问题缺陷原因分析
- 客户问题缺陷非法数据

原因：

- 与系统测试现有页面最接近
- 抽象复用价值最高
- 可率先验证“同构不同口径”的实现模式

### 第二批

- `CC_PRODUCT` 议题
- 延期问题

原因：

- 仍然以表格/查询为主
- 但业务口径和字段面明显独立于系统测试模块

### 第三批

- 缺陷响应效率
- 按功能展示缺陷数量

原因：

- 更偏客户问题域专题统计
- 往往需要独立的统计定义和专题交互

## 6. 第一批页面具体建议

### 6.1 客户问题缺陷汇总

建议做法：

- 复用系统测试缺陷汇总的统计板实现骨架
- 新增客户问题统计板 key
- 行维度仍为模块
- 过滤范围切换为客户问题域的 `milestone / projectId / CC_PRODUCT` 口径

当前进展：

- 已落地第一版，复用统计板链路实现
- 现阶段因本地 `issue_fact` 未形成稳定的 `milestone / CC_PRODUCT` 事实口径，先以“排除系统测试/回归测试范围”作为临时客户问题 scope

### 6.2 客户问题缺陷原因分析

建议做法：

- 复用系统测试缺陷原因分析骨架
- 新增客户问题原因统计板 key
- 保持模块维度与下钻方式一致
- 原因列组可先按新平台事实层实际可支持的粒度落地

当前进展：

- 已落地第一版，复用统计板链路实现
- 当前原因粒度仍受 `issue_fact.reason_category` 现有数据覆盖范围限制

### 6.3 客户问题缺陷非法数据

建议做法：

- 复用系统测试/代码走查类记录页抽象
- 新增客户问题非法数据查询接口和筛选项接口
- 第一版优先保证筛选、分页、展开详情与规则说明

## 7. 风险控制

- 不直接复制系统测试页面文件形成双份维护
- 不用页面名称相同来推导业务口径相同
- 先确认客户问题范围字段，再决定是否需要新的事实层衍生字段
- 第一批优先做“结构同构”的页面，避免一开始就进入专题页高复杂实现

## 8. 交付边界

本计划用于指导客户问题模块后续实现。

它明确了：

- 哪些页面直接复用现有抽象
- 哪些页面只复用底层组件，不直接套现有统计页
- 哪些页面应该优先落地

后续真正开始编码时，应按页面逐个补充对应的统计板设计或记录页接口设计。
