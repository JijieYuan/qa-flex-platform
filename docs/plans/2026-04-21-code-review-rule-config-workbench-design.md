# 2026-04-21 代码走查规则配置工作台设计

## 背景

现有“代码走查非法数据”页面里的规则配置能力，已经验证过字段、条件、预览这条链路可以工作，但仍然存在三个明显问题：

- 规则说明和规则配置混在同一个抽屉里，用户不容易分清“这是解释当前结果”还是“这是我自己在改条件”。
- 规则编辑区更像技术演示，不像面向 QA 的工作台，用户很难一眼理解应该从哪里开始。
- 条件调整后的反馈不够直接，用户不知道改完后到底会筛出什么，也不知道为什么会筛出来。

本轮目标不是扩展企业级规则平台，而是重做一个轻量、易懂、可维护的规则配置工作台。第一期只服务当前用户本人，不影响后端原始数据，也不改变现有规则说明能力。

## 目标

第一期聚焦在“代码走查非法数据”页面，交付一个独立的规则配置工作台，满足以下目标：

- 保留当前规则说明入口和内容，不改其主体结构。
- 删除旧的前端 Demo 式规则配置模块，避免继续维护两套概念。
- 新规则配置工作台作为当前用户自己的高级筛选配置，不修改后端原始记录。
- 用户编辑条件后可以即时看到结果变化、样本记录和变化原因。
- 用户可以保存自己的配置，并将这套配置应用到当前列表。

## 非目标

本轮明确不做：

- 团队共享规则配置
- 规则审批、发布、回滚
- 复杂公式引擎和跨字段加减乘除
- 评审数据管理页面接入
- 重写规则说明接口或规则说明 UI 主体

## 产品边界

需要把“规则说明”和“规则配置”彻底拆开：

- 规则说明：解释系统当前默认规则如何得到结果，属于阅读型内容。
- 规则配置：让当前用户定义自己的条件组合，并即时预览筛选结果，属于操作型内容。

这意味着在“代码走查非法数据”页面里，工具栏上需要同时存在两个入口：

- 规则说明
- 规则配置

其中：

- “规则说明”继续沿用现有抽屉。
- “规则配置”跳转到新的独立页面。

## 页面形态

推荐采用独立路由页，而不是继续放在列表页抽屉中。

原因：

- 规则配置本身已经是一个完整任务，和列表浏览属于两种工作状态。
- 独立页面更容易展示“条件编辑 + 结果预览 + 样本查看”三块内容。
- 后续如果接到其他业务页面，可以复用这套工作台，而不是继续在各页里塞大抽屉。

建议路由形态：

- `/code-review/illegal-records`：代码走查非法数据列表页
- `/code-review/illegal-records/rule-config`：代码走查规则配置工作台

## 页面结构

规则配置工作台采用“三栏工作台”结构：

### 左栏：配置状态

作用是告诉用户“我现在在改什么”和“现在是什么状态”。

内容包括：

- 页面标题与简短说明
- 当前使用范围说明
- 是否存在未保存修改
- 恢复默认配置
- 保存我的配置
- 应用到当前列表

第一期只保留一套“我的配置”，不做多规则集切换。

### 中栏：条件编辑区

作用是让用户用接近自然语言的方式配置条件。

交互单元：

- 字段
- 判断方式
- 值
- 删除条件

支持结构：

- 满足全部条件
- 满足任一条件
- 新增条件
- 新增一组条件

文案避免技术词汇，不出现“表达式树”“节点”“编排”等描述。

### 右栏：结果预览区

这是核心价值区域，需要在用户每次修改后及时反馈：

- 当前会筛出多少条记录
- 占当前数据范围的多少比例
- 比默认规则多了多少或少了多少
- 前若干条样本记录
- 每条样本为什么会出现在这里

右栏必须优先展示结果，而不是先展示规则结构。

## 交互原则

### 1. 先结果，后条件

用户先看到“这套条件会筛出什么”，再去继续微调条件。

### 2. 每次修改都要有反馈

只要字段、判断方式或值发生变化，预览区就要更新，不让用户靠猜。

### 3. 文案要直白

统一使用业务语言：

- 筛出多少条
- 为什么出现在这里
- 保存我的配置
- 应用到当前列表

避免使用容易造成理解负担的词：

- 命中
- 漏斗
- 表达式树
- 节点
- 发布

### 4. 保存与应用分开

- 保存我的配置：把当前用户配置保存到本地
- 应用到当前列表：带着当前配置回到列表页并按此配置显示结果

### 5. 状态显式可见

页面要始终告诉用户：

- 当前配置是否已保存
- 当前列表是否已应用此配置
- 当前预览是否基于最新修改

## 数据与存储方案

### 原始数据

规则配置不修改后端原始记录，只改变筛选结果与展示结果。

### 配置存储

第一期采用本地存储：

- `localStorage`
- 按页面维度存储
- 仅当前用户当前浏览器可见

推荐 key：

- `code-review-illegal-records.rule-config.v1`

本地保存内容包括：

- 配置版本号
- 条件组结构
- 最近保存时间
- 是否启用该配置

### 应用方式

应用到列表时，不直接改原始接口语义，而是把规则配置序列化为页面可识别的查询状态或单独状态容器，再由列表页请求预览接口或读取应用态。

第一期优先保证“规则配置页”和“列表页”的联动可用，不追求通用跨页状态框架。

## 后端方案

第一期不做规则平台，只补一个轻量预览接口即可。

建议新增：

- `POST /api/code-review/illegal-records/rule-config/preview`

请求内容：

- 当前列表基础筛选范围
- 当前规则配置条件

返回内容：

- 当前筛出总数
- 占比
- 与默认规则相比的增减
- 样本列表
- 每条样本的说明文本

说明：

- 后端只负责计算预览结果，不保存配置，不修改原始数据。
- 这样能保证预览结果可信，同时保持实现轻量。

## 旧模块清理范围

需要清理的是“旧规则配置 Demo 模块”，不是规则说明。

计划删除或替换：

- 旧的本地规则配置状态模块
- 旧的规则表达式编辑器
- 旧的代码走查规则 Demo schema
- “代码走查非法数据”页中与旧 Demo 预览相关的状态、计算和样本弹窗

明确保留：

- 规则说明接口
- 规则说明抽屉
- 规则说明相关的后端返回结构

## 前端实现拆分

建议拆成以下模块：

- `CodeReviewRuleConfigView.vue`
  规则配置工作台页面
- `code-review-rule-config-storage.ts`
  本地保存与读取
- `code-review-rule-config-schema.ts`
  当前页面支持的字段定义与默认配置
- `CodeReviewRuleConfigEditor.vue`
  条件编辑区
- `CodeReviewRuleConfigPreview.vue`
  结果预览区

第一期不复用旧规则 Demo 模块，避免新旧概念混杂。

## 测试范围

至少覆盖：

- 旧规则配置模块删除后，代码走查列表页仍可正常打开
- 规则说明入口仍可正常工作
- 规则配置页可正常打开
- 本地保存、恢复默认、应用到列表行为正确
- 预览接口请求与返回能驱动预览区刷新

## 实施顺序

1. 删除旧规则配置 Demo 模块，保留规则说明
2. 在代码走查列表页新增独立“规则配置”入口
3. 新建规则配置工作台页面和本地存储
4. 新增轻量后端预览接口
5. 打通“保存我的配置 / 应用到当前列表 / 返回列表查看结果”
6. 补回归测试
