# 2026-05-21 Issue 事件最终一致性方案

## 背景

内网验证确认 GitLab System Hook 可正常投递到平台，同步日志中已经能看到 `System Hook 唤醒`。但 GitLab System Hook 的触发器不包含 Issue events，创建或修改 issue 不会产生 GitLab System Hook 投递。Issue Hook 属于项目/群组 Webhook 事件，不属于实例级 System Hook。

因此，平台不能再把 System Hook 作为 issue 变化的实时触发来源。Issue、Label、Milestone、Assignee 等业务数据变化应由增量同步和补偿扫描保证最终一致。

## 结论

方案可行，但不能因为单次增量/补偿只耗时约 1 秒，就把它设计成高频全源轮询。平台侧压力较小不等于 GitLab 侧没有压力；频繁读取 GitLab PostgreSQL 仍可能形成类似旧平台爬虫架构的持续负载。

当前可接受的前提是：以分钟级为默认，限定同步表范围，保持同源互斥和去重，并持续观察 GitLab 数据库慢查询、CPU、IO、连接数和锁等待。

建议采用：

1. System Hook 继续作为实例级代码事件唤醒入口，覆盖 push、tag push、merge request、repository update 等 GitLab System Hook 支持的事件。
2. Issue 相关变化不依赖 System Hook，依赖每分钟自动补偿扫描保证最终一致。
3. 默认不做“用户登录即触发同步”，避免多人登录造成额外触发噪声；如果后续需要加速入口，也必须做同源互斥、去重和冷却。
4. 内网把补偿扫描间隔调到 1 分钟。10 秒级只作为短期验证或专项开关，不建议作为默认生产策略。
5. 不恢复“全量表高频轮询”思路。所有自动追数必须限定在业务白名单表和支持增量水位的表上。

## 当前实现基础

- `platform.gitlab-mirror.scheduler-delay-ms` 默认 60000ms，调度器每分钟醒一次。
- `gitlab_sync_configs.compensation_interval_minutes` 最小值为 1 分钟。
- 同一 source 已有运行中的同步时，新的增量/补偿请求会复用或合并，不会并行执行。
- 当前同步日志已能体现 System Hook、增量同步和补偿扫描来源。
- 同步 run 和表任务已记录 `appliedRows`/`rows_applied`，可用于区分“空跑扫描”和“确实写入新数据”。
- 当前事实刷新监听器只在镜像同步成功且 `appliedRows > 0` 时触发事实层刷新，空跑不会造成事实层重建。

## GitLab 侧压力评估

高频增量/补偿对平台自身通常只是短任务，但对 GitLab 服务器的影响取决于源库查询次数和查询范围：

- 如果每 10 秒对大量白名单表执行 `max(updated_at)`、分页游标扫描或多表水位判断，GitLab PostgreSQL 会承受持续查询压力。
- 如果多个用户登录都触发追数，即使平台合并了同源 run，也可能造成频繁排队和重复探测。
- 如果白名单包含大表、缺少可用 `updated_at` 或索引不理想的表，高频补偿会退化成旧式轮询/爬虫压力。
- GitLab 应用本身也依赖同一个数据库；持续外部扫描可能影响 GitLab Web、Sidekiq 和 CI 相关访问。

因此，本方案的安全边界是“低频、增量、白名单、可观测、可熔断”，不是“秒级全库拉取”。

建议上线前至少观察：

- GitLab PostgreSQL CPU、IO、连接数。
- 慢查询日志中是否出现平台同步 SQL。
- 平台每轮增量/补偿的 planned table count、rows scanned、rows applied、duration。
- GitLab 用户侧页面是否出现明显卡顿。
- 同步频率调低/调高时 GitLab 指标变化。

## 推荐运行策略

### 默认内网策略

- `autoSyncEnabled=true`
- `compensationIntervalMinutes=1`
- `platform.gitlab-mirror.scheduler-delay-ms=60000`
- 由后台补偿扫描每分钟自动追数，不在用户登录或页面打开时额外触发同步。
- 若 GitLab 指标有压力，优先把 `compensationIntervalMinutes` 调回 3 到 5 分钟，而不是扩大线程数。
- 若后续要进一步减轻 GitLab 压力，可新增 issue-only 补偿策略，只覆盖 `issues`、`issue_assignees`、`issue_metrics`、`label_links` 等核心表；普通补偿扫描保持更低频。

### 短期验证策略

如果需要验证“近实时”体验，可以临时把调度轮询降到 10 秒，但当前补偿间隔仍以分钟为单位。若要真正支持 10 秒级补偿，需要新增秒级配置或专门的短周期增量触发器，并加入冷却控制。

不建议直接在生产长期启用 10 秒级全源扫描，除非已经完成源库压力、平台数据库写入和事实层刷新频率评估。即便短期验证 10 秒级，也应满足：

- 只对少量核心 issue/MR 相关表生效。
- 单源同一时间仍只允许一个同步 run。
- 连续空跑达到阈值后自动退避到 1 分钟或更长。
- 单轮耗时超过阈值或 GitLab 慢查询增加时自动熔断。
- 有明确开关可立即关闭。

## 增量同步与补偿扫描选择

本方案选择每分钟补偿扫描作为默认后台追数机制，而不是每分钟手动式增量同步。

原因：

- 补偿扫描的语义更适合“后台自动兜底”，不会让用户误以为是人工触发。
- 当前调度器已经按 `scheduler-delay-ms` 和 `compensationIntervalMinutes` 运行，最小补偿间隔正好是 1 分钟。
- 同一 source 的补偿扫描已具备复用/去重逻辑，避免并行执行。
- 增量同步更适合作为用户主动操作或明确加速入口；补偿扫描更适合作为无人值守的最终一致机制。

如果后续仍要增加登录触发增量同步，只能作为“加速追数”，不能绕开同步互斥：

- 只对启用的数据源触发。
- 只在最近同步时间超过冷却窗口时触发，建议冷却窗口 60 秒。
- 如果已有全量、增量、补偿、表刷新或 System Hook run 正在执行，直接复用现有 run，不创建并行 run。
- 日志消息建议使用中文，例如 `用户登录后自动触发增量同步`，方便同步日志排查。
- 触发失败不能阻塞用户登录和页面加载，只记录日志或在运维页面展示。
- 多用户同时登录时只允许产生一次同源追数请求，其余请求应复用、合并或跳过。

## 页面体验策略

每分钟后台同步不能造成业务页面抖动。页面策略如下：

- 后台补偿扫描静默执行，不触发全局 loading，不清空业务表格，不重置分页、排序、筛选或下钻弹窗。
- 镜像设置/同步监控页可以刷新状态和日志，因为它是运维页面。
- 客户问题、系统测试、质量看板、评审数据等业务页面不因后台空跑自动刷新。
- 只有当最新同步 run 的 `appliedRows > 0`，并且相关事实层刷新完成或进入完成态后，业务页面才允许显示轻提示。
- 如果 `appliedRows = 0`，说明只是增量/补偿空跑，不提示用户，也不展示“请刷新”。
- 轻提示应是非阻塞的，例如页面角落或顶部小条：`检测到新数据，可刷新当前页面`。
- 用户点击提示后，只刷新当前页面依赖的数据，不做全平台刷新。
- 不使用频繁 toast，不在每分钟同步开始/完成时弹窗。

这样可以保证后台追数持续运行，但用户正在操作的表格不会每分钟跳动。

## 风险结论

该方案不会天然复现老平台爬虫问题，但如果把频率调到 10 秒、范围扩大到大量表、缺少退避和监控，就可能变成同类问题。

推荐先采用 1 分钟补偿窗口，并在内网用真实 GitLab 指标观察一段时间。只有确认 GitLab 侧无明显压力后，再讨论是否增加登录触发、issue-only 补偿或更短周期的专门增量触发器。

## 2026-05-21 本地验证结果

- 本地 GitLab 真实源已有 3 个项目、389 个 issue、3 个 MR；平台使用直连配置 `sourceInstance=jt`、白名单 12 张核心表完成全量基线同步，run 3 `FULL_SYNC/SUCCESS`，扫描/应用 51599 行。
- 连续触发 5 次手动增量同步时，HTTP 均为 200；其中一次请求被合并到已有同源 run，未创建并行同步。客户问题明细页 `/customer-issues/cc-product-issues` 无路由变化、无全局 loading、表格行数稳定、无 API 5xx 和控制台错误。
- 等待一个 1 分钟自动补偿周期后，产生 run 20 `COMPENSATION/SCHEDULE/SUCCESS`，耗时约 1 秒，`recordCount=0`。页面 80 秒采样期间路由变化 0、全局 loading 0、表格行数稳定、API 5xx 0、请求失败 0、控制台错误 0。
- 因本轮补偿为空跑，页面没有刷新提示，符合“`appliedRows/recordCount=0` 不提示用户、不刷新业务页面”的体验策略。
- 验证发现一个独立边界问题：`sourceInstance=jitter_smoke` 时，增量/补偿 run_id 超过 `sync_runs.run_id varchar(64)`，提交同步返回 500。已记录为 NEW-012。
- NEW-012 已于 2026-05-21 修复：run type 改为短别名，run_id 中的 sourceInstance 可读片段限制为 24 位，并保留 32 位随机串。复测时 `sourceInstance=jitter_smoke` 提交增量同步返回 200，run 29 `INCREMENTAL_SYNC/SUCCESS`，run_id 长度 51。

## 验收标准

1. GitLab 创建 issue 后，即使没有 System Hook delivery，也能在 1 分钟补偿窗口内进入镜像表。
2. 业务事实表刷新后，相关统计或查询页面可看到新增/修改后的 issue。
3. 同步日志中能区分：
   - `System Hook 唤醒`
   - `补偿扫描`
4. `appliedRows = 0` 的补偿空跑不触发业务页面提示，不刷新业务页面。
5. `appliedRows > 0` 且事实层刷新完成后，业务页面只显示轻提示，由用户决定是否刷新当前页面。
6. 多个补偿周期不会产生同源并行同步。
7. 若增量/补偿耗时从 1 秒上升到不可接受范围，应恢复更长补偿间隔，并优先排查大数据量、慢表和事实层刷新。
