# Unified Search Abstraction Design

**Goal:** 为前后端业务搜索建立一套统一的“抽象搜索能力”，支持中文、拼音全拼、首字母、中英混输和去空格模糊匹配，并由多个搜索入口复用，避免重复造轮子。

## Current State

- 前端 `SmartSelect` 已经具备拼音全拼、首字母和英文紧凑匹配能力。
- 普通输入搜索、列表关键字搜索和后端条件筛选仍然是纯文本包含。
- 搜索规则已经出现前端一套、后端另一套的分叉风险。

## Design Decision

采用“共享搜索内核 + 各入口复用”的方案，而不是把所有搜索都强行改造成 `SmartSelect` 组件接口。

原因：

- `SmartSelect` 的公开接口本质上面向“选项筛选”，不适合直接承载自由文本搜索。
- 真正要统一的是“匹配语义”，不是 UI 形态。
- 统一匹配规则后，组件、页面关键字、条件筛选都能复用同一套能力。

## Frontend Design

新增前端通用搜索工具，抽出以下能力：

- 文本归一化
- 去空格紧凑匹配
- 中文转拼音全拼
- 中文转首字母
- 中英混输的连续匹配

前端复用方式：

- `SmartSelect` 直接改为依赖新的通用 matcher
- 以后新增的自由输入前端本地搜索也复用同一工具

## Backend Design

新增后端通用搜索支持类，统一提供：

- 文本规整
- 拼音全拼生成
- 首字母生成
- 多候选 token 组合匹配
- 通用 `contains` 语义

后端先接入这些业务搜索链路：

- `CodeReviewIllegalRecord` 关键字搜索
- `ReviewData` 关键字搜索
- `ReviewData` 条件筛选中的 `contains` / `notContains`

数据库浏览器暂不强行纳入本次范围，因为它是任意表 SQL 搜索，若在本次统一中硬接，会把问题升级成数据库层拼音索引和任意表性能问题。

## Matching Rules

匹配输入与目标文本时，统一支持以下命中方式：

- 原文大小写无关包含
- 去空格后包含
- 全拼包含
- 首字母包含

典型目标：

- `王强`
- `wang qiang`
- `WangQiang`
- `王qiang`

以上输入都应能抽象为同一搜索语义，允许通过 `wq` 命中。

类似地：

- `[草图模块] 算数功能设计说明书评审`

应允许通过 `ct` 命中“草图”，再整体命中整条标题。

## Testing Strategy

前端：

- 为共享 matcher 增加单测
- 确认 `SmartSelect` 仍然支持中文、拼音、首字母和英文混输

后端：

- 为通用搜索支持类增加单测
- 为 `CodeReviewIllegalRecord` 关键字搜索补测试
- 为 `ReviewData` 关键字搜索和 `contains/notContains` 补测试

## Risks

- 中文转拼音在混合字符串里的切分规则要和前端保持稳定。
- `contains/notContains` 一旦统一成抽象搜索语义，测试必须锁住，避免以后只修一侧。
- 任意表数据库搜索如果强行纳入本次，会引入明显的性能和实现复杂度风险，因此暂缓。
