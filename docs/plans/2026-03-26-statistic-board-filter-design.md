# 统计表通用筛选器设计

## 背景

当前统计页已经具备查询、导出、排序、列显示控制、视图持久化等能力，但筛选仍停留在页面特化和简单字段查询层面，无法满足统计分析类表格对“字段自动感知、关系自适应、条件可组合”的统一要求。

本轮目标是在不引入 mock 数据、不改变真实 GitLab 镜像数据来源的前提下，为统计表抽象基类补充一套可复用的通用筛选能力，并先落地到第一页的 `mirror-table-overview` 统计表。

## 目标

1. 前端基于表格字段元数据自动生成筛选项，不允许手动输入字段名。
2. 支持统一的 `字段 + 关系 + 条件值` 配置模式。
3. 支持条件组逻辑：`AND / OR`。
4. 支持数值、文本、时间三类字段的常见筛选关系。
5. 前后端都沉淀到统计表抽象层，后续新统计页可直接复用。

## 方案对比

### 方案一：前后端元数据驱动通用筛选器

由后端返回可筛选字段定义，前端根据字段类型自动渲染对应筛选控件。前端提交统一的条件组结构，后端按真实数据执行过滤。

优点：
- 抽象清晰，适合复用
- 前后端职责边界明确
- 扩展新的字段类型和关系成本低

缺点：
- 首次改动面较大，需要同步调整前后端模型和测试

### 方案二：前端推断字段类型并本地拼装筛选

前端从列定义推断筛选方式，再把最终筛选值拼成普通 query 参数。

优点：
- 改动少

缺点：
- 容易和真实查询能力脱节
- 类型和关系规则散落在前端，后续维护差

### 方案三：每张统计表自己写筛选区

每个业务页自己定义筛选布局和请求结构。

优点：
- 灵活

缺点：
- 无法形成统一抽象
- 后续页面会快速复制粘贴

## 推荐方案

采用方案一：前后端元数据驱动通用筛选器。

## 设计

### 1. 后端筛选元数据

统计表定义中的筛选字段扩展为：
- `key`
- `label`
- `type`
- `placeholder`
- `defaultValue`
- `width`
- `options`
- `operators`

新增支持的字段类型：
- `text`
- `number`
- `datetime`
- `select`

新增支持的关系：
- 文本：`eq`、`ne`、`contains`、`notContains`
- 数值：`eq`、`gt`、`gte`、`lt`、`lte`、`between`
- 时间：`year`、`month`、`day`、`at`、`before`、`after`、`between`
- 选择型：`eq`、`ne`

### 2. 通用筛选请求模型

前端向统计接口提交统一筛选结构：
- 条件组逻辑：`AND | OR`
- 条件项：
  - 字段 key
  - operator
  - value
  - secondaryValue（用于区间）

该结构将以 JSON 字符串参数形式附带到：
- 主表查询
- 明细查询
- 导出查询

### 3. 抽象统计服务

在 `AbstractStatisticBoardService` 中新增通用筛选解析与校验能力：
- 解析 JSON 条件组
- 根据定义校验字段是否合法
- 校验 operator 是否匹配字段类型
- 过滤掉空值条件

各具体统计表服务只负责：
- 定义哪些字段可筛选
- 将已解析的筛选条件转换为真实 SQL 过滤逻辑

### 4. 第一张统计表落地范围

只在 `mirror-table-overview` 这张表上落地：
- 文本字段：
  - `tableName`
- 数值字段：
  - `totalRecords`
  - `withSourceUpdate`
  - `withoutSourceUpdate`
  - `syncedIn24h`
  - `staleSync`
- 时间字段：
  - `lastSyncedAt`
  - `lastSourceUpdatedAt`

注意：
- 这些字段都基于真实镜像表聚合结果生成
- 不引入任何主观业务规则

### 5. 前端抽象表格基类

`StatisticBoardView` 中接入统一筛选栏：
- 以工具栏风格展示，保持页面紧凑
- 支持添加/删除条件
- 支持切换 `AND / OR`
- 字段下拉自动联动 operator 下拉
- 输入控件按字段类型自动切换：
  - 文本：`el-input`
  - 数值：`el-input-number`
  - 时间：`el-date-picker`
  - 选择型：`el-select`

### 6. 导出与明细联动

当前已配置的筛选条件必须同时作用于：
- 主表
- 下钻明细
- CSV 导出

保证用户看到的数据与导出的数据一致。

## 错误处理

1. 非法字段或非法 operator：
- 后端抛出业务异常
- 前端显示统一错误提示

2. 空条件或无效值：
- 前端不提交
- 后端二次兜底忽略

3. 时间区间不完整：
- 前端禁止提交
- 后端兜底校验

## 测试

### 后端
- 条件组解析单元测试
- `mirror-table-overview` 筛选接口测试
- 明细接口带筛选测试
- 导出接口带筛选测试

### 前端
- 筛选器渲染测试
- 字段切换联动 operator 测试
- 数值/文本/时间输入控件切换测试
- `AND / OR` 状态切换测试
- 请求序列化测试

## 成功标准

1. 用户无需输入字段名，仅通过选择字段、关系和值即可完成筛选。
2. 不同字段类型展示对应关系与输入控件。
3. 支持 `AND / OR` 条件组。
4. 主表、明细、导出结果均正确继承同一筛选条件。
5. 能力沉淀在抽象表格基类中，而非当前页面特化。
