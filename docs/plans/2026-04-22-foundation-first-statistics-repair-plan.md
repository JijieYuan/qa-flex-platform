# 统计链路地基优先修复方案

更新时间：2026-04-22  
适用项目：`D:\projects\data_collection_platform`

## 1. 目标

本方案用于从整个项目角度，按“先修地基，再建楼”的顺序修复当前统计链路中的结构性问题。

本轮范围只覆盖：

- `issue` 事实链路
- `merge_request` 事实链路
- `系统测试`
- `客户问题`
- `代码走查`

暂不纳入本轮主修复链路：

- `评审数据`
- `横向对比`

它们放到第二阶段接入，避免当前基础口径还未收口时继续向上扩散复杂度。

## 2. 当前核心问题

### 2.1 页面先行，事实层未完全收口

当前系统测试和客户问题已经开始落页面，但事实层还没有把两类业务域需要的核心字段和范围规则完整沉入。

典型表现：

- `客户问题` 当前临时口径仍是“排除系统测试范围后的 issue”
- 这不等于交接文档中的真实口径：`CC_Product + 2026-01-01 之后创建 + milestone 维度`
- 页面已经能打开，但统计范围还不可信

### 2.2 issue_fact 还没有成为 issue 域唯一可信事实源

`issue_fact` 已经承接了部分系统测试字段，但对客户问题仍不够。

目前已经具备但尚未充分利用或补齐的字段能力包括：

- `reason_category`
- `testing_phase`
- `system_test_label`
- `milestone_title`
- `has_response`
- `is_response_delayed`
- `resolve_deadline_at`
- `is_resolve_delayed`

当前缺口不是“没有表”，而是：

- 字段未完整写入
- 业务 scope 未统一定义
- 归一规则未覆盖客户问题域

### 2.3 业务域边界目前仍混在服务里

现在不少服务内部直接写：

- `inSystemTestScope()`
- `!inSystemTestScope()`

这种写法短期能跑，但会带来三个问题：

1. 客户问题口径变成“非系统测试剩余集”，不是真实业务域
2. 同一个 scope 规则在多个服务中复制
3. 后续接 `CC_Product`、`milestone`、`创建时间下界` 时需要到处改

### 2.4 原因分析空表的根因是事实层命中不足，不是页面没接表

缺陷原因分析当前直接读取 `issue_fact.reason_category`。

所以空表通常来自两类原因：

- 当前范围内没有命中 `reason_category`
- 当前 `reason_category` 归一规则过窄，无法识别老平台模板里的更多原因表达

这不是“少了一张原因表”的问题，而是事实层归一结果不足的问题。

### 2.5 客户问题相关专题规则还停留在文档态

交接文档已经明确了客户问题的核心规则：

- `CC_Product` 范围
- `2026-01-01` 起始时间
- `缺陷调研模板`
- `响应已延期`
- `解决已延期`
- 响应周期 / 解决周期

但这些规则当前尚未形成统一的事实字段和统一的 scope/profile。

## 3. 修复总原则

### 3.1 先统一事实，再统一服务，再推进页面

顺序固定为：

1. 事实层字段与归一规则
2. 业务域 scope/profile
3. 统计/查询服务
4. 页面复用接入

禁止继续先做页面占位替换、再反向补事实层。

### 3.2 一个业务域只允许一个正式 scope 定义

必须显式引入业务域 profile，而不是继续在各服务里散落判断逻辑。

建议正式引入：

- `SystemTestScopeProfile`
- `CustomerIssueScopeProfile`
- `CodeReviewScopeProfile`

每个 profile 只回答四类问题：

- 这个业务域的记录范围是什么
- 默认筛选项是什么
- 哪些事实字段是该域正式口径
- 规则说明如何表达

### 3.3 页面只复用抽象，不复用错误口径

可以继续复用：

- `BaseStatisticTable`
- `BaseRecordTable`
- `StatisticBoardPage`

但不能继续把“错误的临时 scope”当作正式能力扩散到更多模块。

### 3.4 先修已落地模块的可信度，再继续扩子模块

下一阶段不应该先铺更多客户问题专题页，而应先把这两类问题修到可信：

- 已经落地但口径临时的模块
- 尚未落地但直接依赖事实层的模块

## 4. 第一阶段：地基修复

第一阶段只做基础设施，不追求页面数量增长。

### 4.1 建立统一业务域 profile

新增一层 scope/profile 抽象，统一承接业务范围。

建议内容：

- `SystemTestScopeProfile`
  - 范围：系统测试 / 回归测试标签
  - 维度：项目、测试阶段、模块
- `CustomerIssueScopeProfile`
  - 范围：`CC_Product` 议题
  - 约束：`created_at >= 2026-01-01`
  - 维度：`milestone`、模块、紧急程度
- `CodeReviewScopeProfile`
  - 范围：`merge request`
  - 约束：目标分支 `dev`、状态 `MERGED`

目标：

- 彻底去掉各服务中“客户问题 = 非系统测试”的临时定义
- 所有 issue 域服务都从 profile 获取范围

### 4.2 补齐 issue_fact 的客户问题基础字段

优先补齐客户问题第一阶段必要字段：

- `milestone_title` 正式写入
- `project_id / project_name` 对应真实项目归属
- `created_at_source` 统一用于“2026-01-01 起始范围”判断
- `has_response`
- `is_response_delayed`
- `resolve_deadline_at`
- `is_resolve_delayed`
- `illegal_reason`

如果现有字段名已存在，则优先修构建逻辑，不再重复新增列。

本阶段原则：

- 先补事实层写入和归一
- 不新增为页面临时服务的中间结果表

### 4.3 扩展 issue 域模板解析规则

交接文档显示客户问题强依赖“缺陷调研模板”。

因此需要把模板解析从“系统测试非法校验附属逻辑”提升为 issue 域共用能力，至少统一输出：

- 是否按模板回复
- 第一条模板回复时间
- 是否填写预计解决时间
- 预计解决时间
- 是否填写缺陷原因分析
- 是否满足客户问题非法规则

建议做法：

- 抽一层 `IssueTemplateParsingService` 或等价规则组件
- `FactBuildService` 在构建 `issue_fact` 时统一调用

### 4.4 扩展 reason_category 归一规则

原因归一需要从当前“少量稳定 token”升级到“兼容老平台模板表达”的正式规则版本。

目标不是一次性补齐所有老平台细项矩阵，而是先做到：

- 系统测试域不再大面积空原因
- 客户问题域能稳定命中主原因类
- 老模板 / 新模板表达能统一映射到同一事实字段

本阶段先输出：

- 一级原因大类稳定映射
- 未命中原因的观测日志或检查视图

第二阶段再决定是否下沉细粒度子原因字段。

### 4.5 建立事实层验收与回灌机制

必须把“构建成功”与“业务命中成功”分开验证。

第一阶段验收至少包括：

- `issue_fact` 全量重建成功
- 各业务域命中数量可查询
- `reason_category` 非空比例可查询
- 客户问题 `milestone_title` 非空比例可查询
- 模板回复命中数量可查询

建议补一组只用于验收的 SQL / inspector 接口，而不是靠肉眼看页面判断。

## 5. 第二阶段：服务层修复

在地基稳定之前，不继续大面积铺页面。

### 5.1 重构已落地统计服务

优先修以下服务：

- `CustomerIssueDefectSummaryBoardService`
- `CustomerIssueDefectCauseBoardService`
- `SystemTestDefectCauseBoardService`

修复目标：

- 改为消费正式 scope profile
- 改为消费正式事实字段
- Flow/规则说明与交接文档一致

### 5.2 为记录页补统一 issue 查询抽象

当前系统测试查询已经有专用查询服务，但客户问题后续会继续需要查询类页面。

建议抽一层 issue 查询抽象，统一提供：

- 列表查询
- 筛选项查询
- 展开详情
- 导出能力预留

然后分别挂：

- `SystemTestIssueQueryProfile`
- `CustomerIssueIssueQueryProfile`

这样后面实现客户问题非法数据、`CC_PRODUCT` 议题、延期问题时，不需要从零再搭查询骨架。

### 5.3 代码走查服务也要对齐同样结构

虽然本轮不修横向对比，但代码走查属于同一条“先事实、再服务”的链路。

建议同步检查：

- `merge_request_fact` 是否完整承接目标分支、状态、模块、作者、评审时长、缺陷数
- 代码走查服务是否存在口径散落

这样第二阶段接横向对比时，基础才是稳的。

## 6. 第三阶段：按可信度顺序补页面

页面推进顺序不再按“老平台菜单顺序”，而按“底层依赖成熟度”排序。

### 6.1 第一批页面

- `客户问题 -> 缺陷非法数据`
- `系统测试 -> 非法数据`

原因：

- 两者都高度依赖事实层和模板解析
- 一旦这两页能正确跑通，说明 issue 域规则基础已经基本稳住
- 都可以复用 `BaseRecordTable`

### 6.2 第二批页面

- 修正 `客户问题 -> 缺陷汇总`
- 修正 `客户问题 -> 缺陷原因分析`
- 复核 `系统测试 -> 缺陷原因分析`

原因：

- 这几页已经有壳子
- 但可信度目前依赖事实层是否到位

### 6.3 第三批页面

- `客户问题 -> CC_PRODUCT议题`
- `客户问题 -> 延期问题`

原因：

- 它们依赖 milestone、模板解析、响应/解决 SLA
- 在前两批稳定前，不适合先上

### 6.4 第四批页面

- `客户问题 -> 缺陷响应效率`
- `客户问题 -> 按功能展示缺陷数量`
- `系统测试 -> 多元看板`

原因：

- 这些更偏专题页
- 对事实层稳定性和字段覆盖要求更高

## 7. 具体修复顺序

推荐按以下顺序执行，不要跳步：

1. 冻结新增统计页面范围
2. 引入 `ScopeProfile` 抽象
3. 修 `FactBuildService`，补齐客户问题所需字段写入
4. 抽离模板解析能力
5. 扩展原因归一规则
6. 建立事实层验收查询和回灌流程
7. 重构客户问题与系统测试相关服务
8. 落 `客户问题缺陷非法数据`
9. 落 `系统测试非法数据`
10. 修正已落地的客户问题统计页
11. 再推进客户问题专题页

## 8. 需要优先修复的旧实现

以下实现应视为“可运行但不可信”，需要回炉：

- `CustomerIssueDefectSummaryBoardService`
  - 当前 customer scope 仍是临时定义
- `CustomerIssueDefectCauseBoardService`
  - 当前既依赖临时 scope，也依赖薄弱的 `reason_category`
- 客户问题相关路由已接入的真实页
  - 页面壳子可保留
  - 底层统计逻辑必须替换为正式口径

以下实现可暂时保留，只需在第二阶段复核：

- `SystemTestDefectSummaryBoardService`
- `SystemTestPhaseStatisticsBoardService`
- `SystemTestDelayAnalysisBoardService`
- `SystemTestIssueSearchService`

原因：

- 它们已经基于系统测试域工作
- 问题主要不在页面抽象，而在事实层与域 profile 尚未统一

## 9. 风险控制

### 9.1 不在当前阶段继续铺更多客户问题页面

如果继续直接上：

- `CC_PRODUCT议题`
- `延期问题`
- `响应效率`

会把当前临时口径扩散到更多页面，后面返工成本更高。

### 9.2 不新增面向页面的临时结果表

禁止为了解决单页展示问题，额外新增某个“客户问题统计结果表”。

应坚持：

- ODS -> Fact -> Service -> Page

### 9.3 不把交接文档逻辑硬编码散落到多个服务

交接规则应优先沉入：

- `ScopeProfile`
- `FactBuildService`
- 通用规则组件

而不是每个 board service 各写一份。

## 10. 阶段验收标准

### 第一阶段验收

- `issue_fact` 能完整重建
- 客户问题正式 scope 可独立查询
- `milestone_title`、模板回复、SLA、`reason_category` 命中情况可被验证
- 不再使用“客户问题 = 非系统测试”的临时逻辑

### 第二阶段验收

- 客户问题汇总与原因分析改为正式 scope
- 系统测试与客户问题服务都只消费 profile 和 fact
- 规则说明与交接文档一致

### 第三阶段验收

- 非法数据页先完整可用
- 之后专题页按优先级逐步恢复

## 11. 建议的近期落地任务

当前最值得立即开工的不是新的页面，而是以下三件事：

1. 实现 `CustomerIssueScopeProfile`
2. 修 `FactBuildService`，把客户问题关键字段真正沉入 `issue_fact`
3. 抽 issue 域模板解析能力，并补事实层验收接口

这三件完成后，再去做：

- `客户问题 -> 缺陷非法数据`
- `系统测试 -> 非法数据`

成功率最高，返工最少。

## 12. 结论

当前项目的问题不是“前端页面不够多”，而是“事实层和业务域边界还没有成为统一可信底座”。

因此后续推进必须改为：

- 先修事实层
- 再修 scope/profile
- 再修服务
- 最后补页面

只有这样，后续系统测试、客户问题、代码走查三条链路才能共用同一套抽象，而不会继续在业务口径上互相污染。
